# 0001 — Forge.Execution: shared execution contracts

- **Status**: accepted
- **Date**: 2026-05-06
- **Author**: agent

## Context

`Forge.Capability` ships three types that encode the execution outcome vocabulary:
`CapabilityError` (structured error carrier), `CapabilityResult<TResponse>` (discriminated
union `Ok` / `Fail`), and `ICapabilityAspectIriProvider` (aspect-IRI resolution seam, currently
in `Forge.Capability.Http`). A new `Forge.Operations.Http` slice (see Operations.Http ADR-0001)
needs identical primitives for direct entity-CRUD HTTP endpoints that bypass the capability
dispatcher. Duplicating the types would:

1. **Split the platform error vocabulary**: a gateway cannot unify `CapabilityError` and an
   `OperationError` without knowing which slice produced the response.
2. **Bind `Forge.Operations.Http` to `Forge.Capability`**: a transitive dependency on the
   capability dispatcher just to reuse an error record is semantically wrong.
3. **Prevent a uniform correlation model**: both transport slices need to propagate a
   per-execution identifier and a caller-supplied correlation ID; without a shared type the
   correlation envelope would be duplicated per slice.

Additionally, the platform currently has no mechanism to tag an execution with its own
identity or to carry the caller's correlation context. Logs, distributed traces, and error
responses are not correlated across slice boundaries.

## Options

1. **Introduce `Forge.Execution` as a shared, framework-agnostic contract layer.**
   Holds `ExecutionError`, `ExecutionResult<TResponse>`, and `ExecutionCorrelation`.
   No ASP.NET or framework dependency — usable in any transport context (HTTP, messaging,
   in-process). The ASP.NET-specific seams (`IExecutionAspectIriProvider`,
   `HeaderExecutionAspectIriProvider`, correlation middleware, exception-to-422 mapping)
   live in a sibling `Forge.Execution.Http` slice (see ADR-0002).

2. **Duplicate types in each transport slice.**
   Con: two error vocabularies; gateway cannot unify responses; correlation model splits.

3. **Collapse everything into `Forge.Capability`.**
   Con: `Operations.Http` consumers drag the capability dispatcher into their dependency
   graph despite never constructing a `CapabilityDispatcher`; violates single-concern rule.

## Decision

Option 1.

### Dependency graph

```
Forge.Execution          (no framework dependency; net10.0)
Forge.Execution.Http     → Forge.Execution + Microsoft.AspNetCore.App
Forge.Capability         → Forge.Execution   (CapabilityError / CapabilityResult migrate here)
Forge.Capability.Http    → Forge.Execution.Http
Forge.Operations         (no new dependency)
Forge.Operations.Http    → Forge.Execution.Http + Forge.Operations
```

### `ExecutionError`

Replaces `Forge.Capability.CapabilityError`. See Capability ADR-0016.

```csharp
namespace Forge.Execution;

public sealed record ExecutionError(string Code, string Message);
```

### `ExecutionResult<TResponse>`

Replaces `Forge.Capability.CapabilityResult<TResponse>`. See Capability ADR-0016.

```csharp
namespace Forge.Execution;

public abstract record ExecutionResult<TResponse> where TResponse : class
{
    public IReadOnlyList<object> Events { get; init; } = [];

    public sealed record Ok(TResponse Response) : ExecutionResult<TResponse>;
    public sealed record Fail(ExecutionError Error) : ExecutionResult<TResponse>;
}
```

`ICapabilityDispatcher<TCommand, TResponse>.DispatchAsync` changes its return type from
`CapabilityResult<TResponse>` to `ExecutionResult<TResponse>`.

### `ExecutionCorrelation`

Carries the two-tier correlation identity for a single platform execution.

```csharp
namespace Forge.Execution;

public sealed record ExecutionCorrelation
{
    /// <summary>
    /// A unique identifier generated by the platform for this specific execution.
    /// Always non-null; generated via <see cref="Guid.NewGuid"/> at construction time
    /// unless explicitly supplied.
    /// Returned to the HTTP caller as the <c>X-Forge-Execution-ID</c> response header
    /// by <c>Forge.Execution.Http</c> middleware.
    /// </summary>
    public string ExecutionId { get; init; } = Guid.NewGuid().ToString();

    /// <summary>
    /// The correlation ID supplied by the external caller, if any.
    /// Read from the incoming <c>X-Forge-Correlation-ID</c> request header.
    /// <c>null</c> when the caller did not supply a correlation context.
    /// </summary>
    public string? CallerCorrelationId { get; init; }
}
```

#### Correlation header convention

| Header | Direction | Meaning |
|--------|-----------|---------|
| `X-Forge-Correlation-ID` | **Request** | Caller's own correlation identifier; becomes `CallerCorrelationId`. |
| `X-Forge-Execution-ID`   | **Response** | Platform-generated identifier for this execution; returned by middleware. |

The headers follow the platform's `X-Forge-` namespace convention (root ADR-0006 / ADR-0003).

#### Ambient binding

`ExecutionCorrelation` is propagated through the async call stack via an ambient
`AsyncLocal<ExecutionCorrelation?>`, analogous to `AuthorizationContext` in
`Forge.Authorization`. The `Forge.Execution.Http` middleware sets it at the boundary
(see ADR-0002); downstream code reads it from the static ambient accessor rather than
carrying it as a parameter.

```csharp
namespace Forge.Execution;

public static class ExecutionContext
{
    private static readonly AsyncLocal<ExecutionCorrelation?> _correlation = new();

    public static ExecutionCorrelation? Current => _correlation.Value;

    public static IDisposable Use(ExecutionCorrelation correlation) { … }
}
```

## Consequences

- `CapabilityError` and `CapabilityResult<TResponse>` in `Forge.Capability` are replaced
  by `ExecutionError` and `ExecutionResult<TResponse>`; see Capability ADR-0016 for the
  migration plan and backward-compatibility notes.
- All HTTP transport slices share a single error vocabulary; a gateway decodes one
  error schema regardless of which slice produced the response.
- `Forge.Execution` has zero framework dependencies: testable in isolation, usable in
  future non-HTTP transports (messaging, gRPC, CLI).
- Every platform execution is identifiable by `ExecutionId`; the caller's correlation
  chain is preserved in `CallerCorrelationId` for distributed tracing end-to-end.
- Code that held a `CapabilityResult<T>` or `CapabilityError` reference needs a
  one-line `using` update; the discriminated-union shape and record semantics are
  identical.
