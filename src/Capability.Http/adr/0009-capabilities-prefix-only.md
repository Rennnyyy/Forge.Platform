# 0009 — Narrow `MapCapabilities()` to `api/capabilities/` prefix only

- **Status**: accepted; supersedes [0006](0006-route-prefix.md)
- **Date**: 2026-05-06
- **Author**: agent

## Context

Capability.Http ADR-0006 established two route prefixes inside `MapCapabilities()`:

- `api/capabilities/` — for all general capability handlers.
- `api/entities/` — for handlers carrying `[CrudCapabilityHandler]` (generated by
  `Forge.Capability.Generators`).

The dual-prefix logic was introduced to give entity CRUD endpoints a semantically distinct
URL space. With the introduction of `Forge.Operations.Http` (see Operations.Http ADR-0001),
the `api/entities/` URL space is claimed by `MapOperations()` — a separate routing
mechanism that wires entity CRUD directly through `EntityOperations` without the capability
dispatcher pipeline. Two slices registering routes under the same prefix creates:

1. **Collision risk**: a consumer that calls both `MapCapabilities()` and `MapOperations()`
   ends up with competing handlers under `api/entities/`, either failing silently or
   throwing at startup depending on ASP.NET's duplicate-route behaviour.
2. **Wrong ownership**: `api/entities/` is the `Operations.Http` domain; `Capability.Http`
   has no business registering under a prefix it does not own.

Additionally, `Forge.Execution.Http` (Execution ADR-0002) introduces
`IExecutionAspectIriProvider` / `HeaderExecutionAspectIriProvider` as the generalized
replacements for `Capability.Http`'s `ICapabilityAspectIriProvider`. This ADR records
that migration alongside the prefix change.

## Options

1. **Remove the dual-prefix logic. `MapCapabilities()` registers all handlers under
   `api/capabilities/` regardless of whether they carry `[CrudCapabilityHandler]`.
   Migrate `ICapabilityAspectIriProvider` to `IExecutionAspectIriProvider`.**
   `[CrudCapabilityHandler]` remains a valid attribute for capability-layer semantics
   (e.g. future tooling, documentation generation) but is no longer consulted for route
   prefix selection.

2. Keep the dual-prefix logic; accept the collision risk and document that
   `MapOperations()` and `MapCapabilities()` must not be called together for the same
   entity type. Con: implicit constraint, not enforced; brittle; wrong ownership.

## Decision

Option 1.

### Changes to `EndpointRouteBuilderExtensions`

The `isCrud` / prefix-branching logic is removed:

```csharp
// Before (ADR-0006)
var isCrud    = descriptor.HandlerType.GetCustomAttribute<CrudCapabilityHandlerAttribute>() is not null;
var prefix    = isCrud ? "api/entities" : "api/capabilities";
var routePath = $"{prefix}/{attr.Identity.ToRoutePath()}";

// After (this ADR)
var routePath = $"api/capabilities/{attr.Identity.ToRoutePath()}";
```

The `CrudCapabilityHandlerAttribute` import is no longer needed in this file.

### Migration of `ICapabilityAspectIriProvider`

`Forge.Capability.Http` currently defines and registers its own
`ICapabilityAspectIriProvider` / `HeaderCapabilityAspectIriProvider`. Both are replaced
by the shared `IExecutionAspectIriProvider` / `HeaderExecutionAspectIriProvider` from
`Forge.Execution.Http`.

| Before | After |
|--------|-------|
| `ICapabilityAspectIriProvider` | `IExecutionAspectIriProvider` |
| `HeaderCapabilityAspectIriProvider` | `HeaderExecutionAspectIriProvider("X-Forge-Capability-AspectIri")` |
| `RegisterEndpoint<T,R>` inline try/catch | `ExecutionEndpointHelper.InvokeAsync(…)` |

`AddCapabilityHttp()` registers:

```csharp
services.TryAddSingleton<IExecutionAspectIriProvider>(
    _ => new HeaderExecutionAspectIriProvider("X-Forge-Capability-AspectIri"));
```

`ICapabilityAspectIriProvider` is deleted from `Forge.Capability.Http`. Applications
that registered a custom `ICapabilityAspectIriProvider` must re-register against
`IExecutionAspectIriProvider`.

`Forge.Capability.Http` adds a `ProjectReference` to `Forge.Execution.Http` and
removes `Forge.Aspects` from its direct references (the exception catch types are now
supplied transitively via `Forge.Execution.Http`).

## Consequences

- `MapCapabilities()` is simpler: one prefix, no branching.
- `[CrudCapabilityHandler]` remains in `Forge.Capability` as a semantic marker; its
  routing role is retired. Future tooling or documentation generators may still read it.
- `api/entities/` is exclusively owned by `Forge.Operations.Http`'s `MapOperations()`.
- The aspect-IRI header for capability dispatch remains `X-Forge-Capability-AspectIri`;
  the provider implementation changes, not the header name.
- Any application that exposed `[CrudCapabilities]`-generated handlers at
  `api/entities/…` must update its client URLs and Bruno collections to
  `api/capabilities/…`. This is a breaking URL change and must be communicated to
  consumers prior to deployment.
