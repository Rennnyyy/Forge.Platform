# 0019 — Platform managed-entity HTTP layers must wire aspect enforcement

- **Status**: accepted
- **Date**: 2026-05-12
- **Author**: agent

## Context

ADR-0017 established that any HTTP layer for a platform-managed entity (one whose
endpoints are hand-written rather than generated by `[OperationEndpoints]`) must use the
shared `Forge.Operations.Http` response contracts. It did not address the *aspect*
contract: whether such a layer must also accept and enforce `IOperationAspect`
validation on its write transactions.

Generated endpoints (via `MapOperations()`) automatically:
1. Read the `X-Forge-Operation-AspectIri` request header.
2. Pass the IRI to every CUD `EntityTransaction` operation.
3. Reach `AspectEnforcingTransactionalStore` via the unkeyed `ITransactionalEntityStore`
   registered by `AddForgeAspects()`.

Hand-written layers for managed entities — beginning with `Forge.Branch.Http` — use
a **keyed** store (`"forge.branch.management"`). `AddForgeAspects()` only decorates
the *unkeyed* `ITransactionalEntityStore`, so the keyed management store escapes aspect
enforcement entirely. There is also no convention requiring the endpoint handler to read
the aspect header. The combined effect: callers cannot apply `IOperationAspect`
SHACL constraints to branch or snapshot writes.

## Decision

**Any hand-written HTTP layer for a platform-managed entity MUST satisfy all four of the
following obligations:**

### 1. Shared response contracts
Use `OperationCreatedResponse`, `OperationUpdatedResponse`, `OperationDeletedResponse`,
and `OperationListResponse<T>` from `Forge.Operations.Http` (established by ADR-0017).

### 2. Aspect enforcement on the backing store
The DI helper for the slice's HTTP layer (e.g. `AddForgeBranchHttp()`) MUST call
`services.AddForgeAspectsForKeyedStore(storeKey, sparqlStoreResolver)` from
`Forge.Aspects.DependencyInjection`, which inserts an `AspectEnforcingTransactionalStore`
decorator at the top of the keyed store's chain.

`AddForgeAspectsForKeyedStore` requires a `Func<IServiceProvider, ISparqlQueryStore>`
that resolves the raw backend before guard decorators are applied, because SPARQL
queries used by the Context pass must target the unguarded backend directly.

The raw backend is exposed by `AddForgeBranch()` as a keyed `ITransactionalEntityStore`
under the key `"<storeKey>.raw"` (e.g. `"forge.branch.management.raw"`), and the HTTP
DI helper casts it to `ISparqlQueryStore`.

### 3. Aspect IRI threading
Every CUD endpoint handler MUST read `X-Forge-Operation-AspectIri` using
`HeaderExecutionAspectIriProvider` and pass the resolved IRI to all
`EntityTransaction.Create / Update / Delete` operations and to any application-service
methods that issue management transactions (e.g. `BranchSeedingService`).

A `null` resolve (absent header) falls back to `Aspect.NoOpIri` — identical behaviour
to generated endpoints.

### 4. Managed-entity store key registration
`AddForgeBranch()` (the *entity*-layer DI helper, not the HTTP helper) MUST register a
`ManagedEntityStoreKeyRegistration(storeKey)` in the DI container. This record lives in
`Forge.Repository.DependencyInjection`.

`AddForgeAspectsForKeyedStore(storeKey, …)` registers a matching
`AspectEnforcedKeyedStoreRegistration(storeKey)`.

A `ManagedEntityAspectValidationService` (hosted service in `Forge.Aspects`) validates
at startup that every `ManagedEntityStoreKeyRegistration` has a matching
`AspectEnforcedKeyedStoreRegistration`. If `AddForgeAspects()` is not active (no
`IAspectStore` registered) the service is not registered and no validation runs — callers
that do not use aspects at all are not affected.

## Options considered

### Keyed-store SPARQL resolution

**Option α** — `AddForgeBranch()` registers a separate keyed `ISparqlQueryStore`
for the raw backend.
Pro: no changes to `AspectEnforcingTransactionalStore`.
Con: `AddForgeBranch()` must know it may be paired with `AddForgeAspects()`, leaking
an Aspects concern into the entity layer.

**Option β** — `AddForgeAspectsForKeyedStore` accepts a `Func<IServiceProvider, ISparqlQueryStore>`
resolver. `AddForgeBranch()` exposes the raw backend under `"<key>.raw"` as an
`ITransactionalEntityStore`; the HTTP helper provides the cast.
Pro: `AddForgeBranch()` is Aspects-agnostic; the SPARQL resolution strategy is declared
at the HTTP layer where the coupling is already present.
Con: slight indirection in the factory delegate.

**Decision: Option β.**

### Startup validation placement

**Startup filter** (runs before first request): catches the misconfiguration at HTTP
pipeline build time; visible to integration tests that do not send requests.

**Hosted service** (runs at `IHostedService.StartAsync`): runs before the application
starts serving, so the misconfiguration is caught during startup.

**Decision: Hosted service** in `Forge.Aspects` (`ManagedEntityAspectValidationService`),
conditional on `IAspectStore` being present.

## Consequences

- `Forge.Branch` gains a `ManagedEntityStoreKeyRegistration` registration and exposes
  a `"forge.branch.management.raw"` keyed store.
- `Forge.Aspects` gains `AddForgeAspectsForKeyedStore` and `ManagedEntityAspectValidationService`.
- `Forge.Branch.Http` gains `AddForgeBranchHttp()`, which is the single call that
  satisfies obligations 2–3 and kicks off the startup validation check.
- Future platform-managed entity slices (e.g. any `Authorization.Http` entity surface)
  must follow the same four-obligation pattern. Obligation 4 ensures the startup
  validator will catch omissions automatically whenever `AddForgeAspects()` is active.
- `BranchSeedingService.CreateSnapshotAsync` and `DeleteSnapshotAsync` gain an explicit
  `string aspectIri` parameter so the IRI threads from the HTTP handler through to
  the management `EntityTransaction`.
