0001 Single-solution monorepo with src/ and tests/ Accepted 2026-04-29

Establish a single .slnx solution containing all library packages in src/ and all test projects in tests/, with Directory.Build.props and Directory.Packages.props at the monorepo root for shared MSBuild settings and central package version management.

0002 .NET 10 + Central Package Management Accepted 2026-04-29

Target net10.0 for all runtime packages and netstandard2.0 for Roslyn source generators. Use Central Package Management (CPM) via Directory.Packages.props to ensure a single canonical version per NuGet dependency across the entire monorepo.

0003 Architecture Decision Records as the design source of truth Accepted 2026-04-29

Every non-trivial design choice must be captured in a numbered ADR in the adr/ folder before implementation begins. ADRs are the authoritative record of the why, not just the what, for every architectural decision in the platform.

0004 Default namespace = Forge.<library> Superseded by 0008 2026-04-29

Establishes the Forge.<LibraryName> namespace convention for all packages, mapping directly to the folder name under src/. Superseded by ADR-0008 which removed the Entity. prefix from satellite package names.

0005 RDF-backed Entity Repository as a separate slice Accepted 2026-04-30

The persistence layer uses RDF named graphs rather than a relational schema, with dotNetRDF as the in-process implementation. The repository slice is isolated so that alternative backends (GraphDB, future SPARQL endpoints) can be substituted by implementing IEntityStore without touching domain code.

0006 Canonical base URL is https://forge-it.net Accepted 2026-05-01

All entity IRIs are minted under https://forge-it.net as the base. This is configurable via EntityOptions.BaseIri but defaults to the canonical URL. Predicate IRIs use https://forge-it.net/predicates/{predicatePath}/{name}.

0007 Shared test fixtures project for sample entities Accepted 2026-05-01

A tests/Entity.Tests.Fixtures project holds shared entity definitions and DI setup used by multiple test suites, preventing duplication. Companion ADR-0015 later split this into Core and full-CRUD tiers.

0008 Remove Entity. prefix from satellite package names Accepted 2026-05-02

Satellite packages that were originally named Forge.Entity.* (e.g. Forge.Entity.Operations) are renamed to drop the Entity. prefix (e.g. Forge.Operations). The feature namespace follows the package name convention. Supersedes ADR-0004's implicit assumption about satellite naming.

0009 ADR adjustment policy: inline notes for identifier-only changes Accepted 2026-05-03

When a subsequent decision changes only an identifier (type name, method name, namespace) established by an earlier ADR, an inline note is added to the original ADR rather than creating a new numbered record, reducing ADR proliferation for cosmetic renames.

0010 Sub-folder structure inside slice directories Accepted 2026-05-03

Each slice directory under src/ organises its files into sub-folders: DependencyInjection/, Transaction/, Mapping/, Rdf/, etc. This keeps root directory clean and establishes a predictable navigation pattern across all packages.

0011 Samples folder for runnable demonstration applications Accepted 2026-05-04

A top-level samples/ folder contains runnable ASP.NET Core applications that demonstrate platform features end-to-end. The sample application depends on all src/ packages and is part of the .slnx solution but not published as a NuGet package.

0012 Integration-testing samples via Bruno CLI and a subprocess host Accepted 2026-05-04

The tests/Application.Sample.Tests project drives the committed Bruno HTTP collection against a live sample-app process spawned as a subprocess. The test class uses xUnit [Fact]s to gate each Bruno chapter, giving CI visibility per story chapter.

0013 Bruno collection organised as story chapters Accepted 2026-05-04

The Bruno collection under samples/Application.Sample/bruno/ is structured as numbered story chapters (e.g. 01-books/, 02-branches/). Each chapter exercises one feature slice end-to-end. Amended by ADR-0016 to expand to 14 chapters.

0014 Registration-order-independent DI decorator chain Accepted 2026-05-09

The keyed-service decorator chain (Backend → AspectEnforcing → EventEmitting → Guard) is built using deferred factory delegates that resolve the next-in-chain at provider build time. This means AddForgeAspects(), AddForgeEntityEvents(), etc. can be called in any order in Program.cs without breaking the chain.

0015 Split entity test fixtures into Core and full-CRUD projects Accepted 2026-05-04

tests/Entity.Tests.Fixtures.Core holds only entity definitions and minimal DI wiring, while tests/Entity.Tests.Fixtures adds generated-CRUD methods and full operation endpoint setup. Downstream test projects reference only the tier they need to minimise compile-time coupling.

0016 Bruno collection expanded to 14 chapters Accepted 2026-05-04

The Bruno collection is expanded from its initial scope to 14 story chapters covering all major feature slices: books, branches, snapshots, capabilities, aspects, data-records, object-bearing entities, messaging diagnostics, async capability dispatch, studios, recordings, genres, featured-artists, and track-masters. Amends ADR-0013.

0017 Platform entity HTTP layers must use shared Forge.Operations.Http contracts Accepted 2026-05-12

Any hand-written HTTP layer for a platform-managed entity (e.g. Forge.Branch.Http) must use the shared response types from Forge.Operations.Http — OperationCreatedResponse, OperationUpdatedResponse, OperationDeletedResponse, OperationListResponse<T> — rather than defining its own response records.

0018 Bruno integration tests also run against GraphDB backend Accepted 2026-05-12

A parallel BrunoGraphDbIntegrationTests class in Application.Sample.Tests backs the full Bruno suite with a live GraphDB container (via BrunoGraphDbFixture). All 15 test cases use [SkippableFact] and skip gracefully when no container runtime is available.

0019 Platform managed-entity HTTP layers must wire aspect enforcement Accepted 2026-05-12

Any hand-written HTTP layer for a platform-managed entity must satisfy four obligations: use shared response contracts, call AddForgeAspectsForKeyedStore() to wire aspect enforcement on the keyed management store, thread X-Forge-Operation-AspectIri through every write, and register a ManagedEntityStoreKeyRegistration. A startup-time ManagedEntityAspectValidationService enforces this automatically when AddForgeAspects() is active.

0020 Messaging abstractions slice Accepted 2026-05-13

Introduces three new slices: Forge.Messaging.Abstractions (broker-agnostic interfaces and MessageEnvelope<T>), Forge.Messaging.Kafka (Confluent.Kafka implementation), and Forge.Messaging.InMemory (in-process channel-based implementation for tests). Feature slices depend only on the abstractions; no Kafka reference leaks into domain slices.

0021 Entity change event stream Accepted 2026-05-13

An EventEmittingEntityStore decorator sits above AspectEnforcingTransactionalStore in the decorator chain. After each valid write it publishes an EntityChangedEnvelope<TDto> to both a history topic (forge.entities.{type}.history) and a compacted state topic (forge.entities.{type}.state). Delivery is at-least-once; consumers must be idempotent.

0022 Async capability and operations command bus Accepted 2026-05-13

Forge.Capability.Messaging adds IAsyncCapabilityDispatcher<TCommand,TResponse> supporting fire-and-forget (PublishAsync) and request-reply (PublishAndWaitAsync). The in-process ICapabilityHandler logic is reused unchanged on the consumer side; a PendingReplyRegistry correlates replies by ExecutionId.

0023 Object Storage slices Accepted 2026-05-13

Three new slices address binary object storage: Forge.ObjectStorage.Abstractions (provider-agnostic IObjectStore), Forge.ObjectStorage.InMemory (in-memory blob store for tests), and Forge.ObjectStorage.Http that wires eight REST routes per [ObjectBearing] entity via MapObjectOperations() — covering metadata CRUD, streaming upload, binary download, and blob-only delete.
