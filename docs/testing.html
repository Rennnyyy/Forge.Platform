Testing

Philosophy, tooling, shared fixtures, and patterns for unit, integration, and end-to-end testing across the Forge.Platform monorepo.

Philosophy

Real behaviour, fake infrastructure

Unit tests run against in-memory backends (InMemoryEntityStore, InMemoryMessageBroker). All business logic exercises real DI-wired services — no hand-rolled mocks for domain code.

Shared fixtures, not shared state

Re-usable seed data lives in Entity.Tests.Fixtures.Core (entity definitions) and Entity.Tests.Fixtures (repository-backed populated stores). Each test class receives a fresh fixture instance — never static state. See ADR-0007 and ADR-0015.

Bruno for integration tests

HTTP integration tests are written as Bruno collections in spec/. Each chapter maps to one behavioural story defined in ADR-0013 and expanded in ADR-0016. Tests can run against GraphDB (ADR-0018).

One test project per slice

Every src/ slice has a matching tests/ project. Generator test projects use Microsoft.CodeAnalysis.Testing to verify roslyn source output at compile time.

Unit Tests — InMemory Backends

Swap the real repository backend for an in-memory one in your WebApplicationFactory or test DI container:

// Service registration in test host:
services.AddForgeEntityRepository(config)
        .RegisterEntity<Book>()
        .UseInMemory();        // <-- in-memory RDF store, no external infra needed

services.AddForgeMessagingInMemory(); // in-memory Kafka analogue

Key in-memory types

TypePackagePurpose
InMemoryEntityStore Forge.Repository.InMemory dotNetRDF TripleStore backed by a MemoryStoreManager. All SPARQL and update operations are evaluated in-process.
InMemoryMessageBroker Forge.Messaging.InMemory Channel<T>-backed broker. Implements both IMessageProducer and IMessageConsumer and delivers messages synchronously within the same process.

Resetting state between tests

// Implement IAsyncLifetime in your test class (xUnit) to get a fresh store per test:
public class BookTests : IAsyncLifetime
{
    private readonly WebApplicationFactory<Program> _factory;
    private HttpClient _client = null!;

    public BookTests()
    {
        _factory = new WebApplicationFactory<Program>()
            .WithWebHostBuilder(b => b.ConfigureServices(ConfigureTestServices));
        _client = _factory.CreateClient();
    }

    private static void ConfigureTestServices(IServiceCollection services)
    {
        services.AddForgeEntityRepository(/* test config */)
                .RegisterEntity<Book>()
                .UseInMemory();
    }

    public async Task InitializeAsync() => await Task.CompletedTask;
    public async Task DisposeAsync()    => await _factory.DisposeAsync();
}

Shared Test Fixtures

The fixture split (ADR-0015) separates compile-time-only entity definitions from heavier repository-backed fixtures to keep non-repository test projects lean.

ProjectPurposeUsed by
Entity.Tests.Fixtures.Core Entity class definitions (TestEntity, TestRelated, etc.) that compile without any repository dependency. Safe to reference from generator tests and any slice-level unit test. Entity.Generators.Tests, Operations.Generators.Tests, Sparql.Tests
Entity.Tests.Fixtures Pre-populated in-memory stores using Repository.InMemory. Provides EntityFixture, RelationshipFixture, and BranchFixture with seed data for common test scenarios. Entity.Tests, Repository.InMemory.Tests (via Tests/), Branch.Tests, etc.

Using EntityFixture

// In xUnit — inject fixture via IClassFixture<T>:
public class MyTests : IClassFixture<EntityFixture>
{
    private readonly EntityFixture _fixture;

    public MyTests(EntityFixture fixture) => _fixture = fixture;

    [Fact]
    public async Task CanReadSeededEntity()
    {
        var book = await _fixture.Books.ReadAsync(_fixture.SeedBookIri);
        Assert.NotNull(book);
    }
}

Integration Tests

Application.Sample.Tests

Standard xUnit WebApplicationFactory integration tests that boot the full Application.Sample pipeline against an in-memory backend. These tests verify HTTP contract compliance, response codes, and entity lifecycle.

Bruno API tests

The spec/ folder holds a Bruno collection with 14 chapters (ADR-0016). Each chapter is a self-contained story directory with a .bru file per request:

spec/
  forge/
    01-create-book/
      01-create.bru
      02-read-back.bru
    02-update-book/
      01-update.bru
    …
    14-object-storage/
      01-upload-blob.bru
      02-download-blob.bru

Run via Bruno CLI:

bru run spec/forge --env local

GraphDB backend tests

Set FORGE_GRAPHDB_BASEURL and FORGE_GRAPHDB_REPOID environment variables and run with the graphdb Bruno environment to execute tests against a real Ontotext GraphDB instance (ADR-0018):

bru run spec/forge --env graphdb

Testing Roslyn Source Generators

Generator test projects (Entity.Generators.Tests, Operations.Generators.Tests) use the Microsoft.CodeAnalysis.CSharp.Testing harness. Each test compiles an input source string and asserts on the generated output:

// Example generator test (Entity.Generators.Tests):
[Fact]
public async Task GeneratesRdfMapper()
{
    var source = """
        using Forge.Entity;

        [Entity(Path = "widgets")]
        [Identity(IdentityKind.Random)]
        public partial class Widget { public string Name { get; set; } = ""; }
        """;

    await new CSharpSourceGeneratorTest<EntityMapperGenerator, DefaultVerifier>
    {
        TestState =
        {
            Sources        = { source },
            GeneratedSources = { (typeof(EntityMapperGenerator), "Widget.Mapper.g.cs", expectedOutput) }
        }
    }.RunAsync();
}
Generator tests reference Entity.Tests.Fixtures.Core (not Entity.Tests.Fixtures) to avoid pulling in Repository.InMemory which would create a circular in generator test compilation.
