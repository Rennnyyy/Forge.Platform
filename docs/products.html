Products

Information Framework Product

A domain-first runtime type system with pluggable RDF persistence, operational semantics, and built-in contract enforcement. Model your domain as typed entities; swap persistence backends without touching domain code.

  • 🧱
    Entity Core
    Typed entities with IRI-based identity, EntityRef, EntityCollection, EntityOptions, and EntitySession lifecycle management. Source generators eliminate boilerplate identity declarations.
  • 🗄️
    Persistence feature
    IEntityRepository<T> over an IEntityStore abstraction. Reflection-based RDF mapper; Create / Replace write semantics; ordered collection support via rdf:List.

    Feature options — choose one or both backends:
    In-Memory GraphDB
  • ⚙️
    Operations & Transactions
    EntityOperations model entity mutations as first-class typed operations, including transactional semantics. Source generators wire operation handlers at compile time, keeping runtime overhead near zero.
  • ✅
    Aspects & Contract Enforcement
    Three validation legs: Write (CUD operations — SHACL + SPARQL context), Query (filter gate + result-graph SHACL), and Message (command/response shape contracts). TTL-encoded shapes; structured AspectViolation payloads. Also consumed by the Capability Framework via the shared aspect interfaces.
  • 🔍
    SPARQL Query Layer Phase 2
    Future slice (Forge.Sparql): query model, emitter, rewriter pipeline for security filters, federation, and property-path expansion. IEntityStore already accepts opaque SPARQL queries today — the seam is in place.

Capability Framework Product

A zero-ceremony command/response dispatcher. Define a capability, annotate a handler — get in-process dispatch, HTTP endpoints, and aspect-enforced message contracts without boilerplate.

  • ⚡
    Capability Core
    ICapabilityDispatcher with strongly-typed ICapabilityHandler<TCommand, TResponse>. [Capability("identity.string")] attribute drives identity, routing, and queue names. CapabilityResult encapsulates success / structured errors.
  • 🌐
    HTTP Transport feature
    app.MapCapabilities() scans all registered handlers at startup and registers Minimal API endpoints automatically. Routes are derived from capability identity strings. Verb selection: attribute → override → POST. Query-string binding for GET / DELETE; JSON body for POST / PUT / PATCH.
  • 📬
    Per-Request Aspects
    ICapabilityAspectIriProvider resolves IMessageAspect instances from X-Forge-* HTTP headers before dispatch. Pluggable — replace the default header reader with any strategy. Absent headers → permissive (null); fail-closed mode configurable.
  • ✅
    Message Aspect Enforcement via IF
    SHACL shape validation applied to command and response objects before and after dispatch via the IMessageAspect leg — the Message validation leg defined in the Information Framework. No duplication; the Capability Framework takes a dependency on Forge.Aspects.Abstractions only.
🔒

Authorization — Internal Variance Point

Authorization is not a customer-facing product option. It is a resolved variance point: the concrete authorization strategy is selected at deployment/staging time and baked into the service configuration before the product is handed to a consumer. Both products share the same authorization seam (IAspectGuard / Forge.Authorization) and the HTTP agent-token middleware (Forge.Authorization.Http). The concrete guard wraps the Aspect engine from the Information Framework.

Anonymous No identity required — open access
No-Op Guard Allow-all — development & CI
Full (JWT / Agent Token) Production — header extraction + guard
